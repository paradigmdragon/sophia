

1. 전제 정리 (중요)

이 UI 설계는 아래를 절대 침범하지 않는다.
	•	❌ 글 흐름을 끊지 않는다
	•	❌ 분석 결과를 강요하지 않는다
	•	❌ 수정·개선 지시를 하지 않는다
	•	❌ 점수·경고음·빨간줄 같은 ‘채점 UX’ 금지

Sophia는 **편집기 위에 얹힌 “사고 보조 레이어”**다.

⸻

2. 원고모드에서 Sophia의 위치

기본 상태
	•	Sophia는 꺼져 있어도 된다
	•	기본값은 ON이 아니라 PASSIVE

즉,
	•	사용자가 글을 쓰고
	•	Sophia는 조용히 감지만 한다

⸻

3. UI 구성 요소 4개 (최소)

① 상단 툴바: Frame 토글 (아주 작게)

[ 원고 ]   [ Sophia ▽ ]
Sophia ▽ 클릭 시 드롭다운

✓ Epidora (사고 점검)
□ Logical (논증)
□ Bias (인지)
□ System (시스템)

	•	Epidora만 기본 체크
	•	복수 선택 가능
	•	기술 용어 설명 없음 (툴팁으로만)

⸻

② 본문 옆: “사고 신호 마커” (가장 중요)

문장 오른쪽 여백에 아주 작은 점만 표시
	•	🟣 Epidora
	•	🔵 Logical
	•	🟢 Bias
	•	🟠 System

조건:
	•	아이콘은 1~2개만
	•	과다 표시 시 자동 축약
	•	신호가 없으면 아무것도 안 보임

👉 사용자는 ‘문제가 있다’는 느낌만 받지
‘무슨 문제인지’는 강요받지 않는다

⸻

③ 마우스 오버: “설명”이 아니라 “관측”

아이콘에 마우스를 올리면 이렇게 뜬다.

사고 점검 (Epidora)
이 문장에서
정의되지 않은 개념이
고정된 실체처럼 사용될 가능성

여기까지가 전부다.
	•	❌ 틀렸다
	•	❌ 오류다
	•	❌ 고쳐라
없음.

⸻

④ 클릭 시: 질문 카드 (Sophia의 핵심 UX)

아이콘 클릭 → 작은 카드가 옆에 뜬다.

사고 점검 질문

• 이 개념은 언제 정의되었나요?
• 이 단어는 과정인가요, 대상인가요?

[ 닫기 ]   [ 질문만 남기기 ]

선택지는 딱 두 개다.
	•	닫기
→ 아무 기록 없음
	•	질문만 남기기
→ 문서 하단 “사고 질문 영역”에 추가

⸻

4. 문서 하단: 사고 질문 영역 (선택적)

문서 맨 아래, 접혀 있는 섹션

▸ 이 글에서 남은 사고 질문 (3)

열면:
	•	“이 개념은 언제 정의되었는가?”
	•	“이 판단의 관찰 시점은 어디인가?”
	•	“이 결론에 도달한 과정은 무엇인가?”

이 영역은:
	•	체크리스트 ❌
	•	할 일 ❌
	•	숙제 ❌

👉 **“생각의 여백”**이다.

⸻

5. SonE와의 연결 (아주 느슨하게)

사고 질문 옆에 작은 점 메뉴

⋮

클릭 시:
	•	SonE로 질문 보내기
	•	토론으로 열기
	•	그냥 두기

⚠️ 기본값은 아무것도 안 함

Sophia는 행동을 유도하지 않는다.

⸻

6. 왜 이 UX가 좋은가 (핵심 이유)

1) 글의 리듬을 깨지 않는다
	•	자동 팝업 ❌
	•	실시간 강조 ❌
	•	분석 중단 ❌

2) “비판받는 느낌”이 없다
	•	경고 ❌
	•	오류 ❌
	•	점수 ❌

3) 쓰는 사람이 주도권을 가진다
	•	볼지 말지 선택
	•	질문을 남길지 선택
	•	무시해도 페널티 없음

⸻

7. 기존 Logos64/분류 시스템과의 결정적 차이

기존 상상	Sophia UX
개념에 태그 부착	문장 옆 사고 신호
지식 그래프	사고 질문 누적
분류 정확도	사고 자각
AI가 정리	사용자가 생각

👉 이게 LLM 시대에도 살아남는 이유다.

⸻

8. v0에서 반드시 지켜야 할 제약

이걸 깨면 실패한다.
	1.	한 문장에 질문 최대 2개
	2.	아이콘 항상 작게
	3.	자동 수정 제안 절대 금지
	4.	Sophia 결과는 문서 품질 평가에 사용 금지

⸻

9. 여기까지의 결론
	•	Sophia는 기능이 아니라 태도
	•	원고모드에 “붙어있는 사고 보조 레이어”
	•	에피도라는 비판 도구가 아니라 관측 렌즈
